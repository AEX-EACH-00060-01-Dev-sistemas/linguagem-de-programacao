## Compilação x Interpretação

No mundo da computação, o computador entende apenas uma linguagem muito simples: 0 e 1, o estado "desligado" e o "ligado". Através de convenções impostas nessa área, é possível representar qualquer coisa por meio desses 0s e 1s e realizar as operações fundamentais para esse feito. 
\
O *bit* (a linguagem de 0 e 1) é a peça fundamental para os computadores, mas é muito difícil de ser trabalhado diretamente por humanos. Abaixo está um exemplo de como uma frase simples fica gigante quando convertida para *bits*:

* **Texto**: "Olá querido espectador!"
* **Bits**: `01001111 01101100 11100001 00100000 01110001 01110101 01100101 01110010 01101001 01100100 01101111 00100000 01100101 01110011 01110000 01100101 01100011 01110100 01100001 01100100 01101111 01110010 00100001`

Já deu para perceber que trabalhar diretamente com 0 e 1 seria caótico. Por isso, diversos pesquisadores criaram soluções para nos permitir desenvolver sistemas de maneira mais fácil. Assim, surgiram as primeiras linguagens de programação, com destaque para o *Assembly*, que simplificou muito o trabalho na época.
\
Depois de anos de evolução, surgiram linguagens que facilitaram ainda mais os processos de criação. Neste momento, veremos um dos pontos fundamentais que influenciam diretamente o desempenho e o funcionamento delas.

### Compilação
A primeira forma de traduzir o nosso código para a linguagem de máquina é através da **compilação**. Isso é feito por um programa chamado *compilador*. Ele é responsável por pegar todo o código escrito por você e traduzi-lo de uma vez só para um arquivo binário (em *bits*), gerando o programa executável.
\
Esse método traz grandes vantagens em relação ao **desempenho**, por já entregar o código direto na linguagem nativa do processador, e também ajuda na **detecção de erros antes da execução** (como erros de sintaxe ou de tipo).

> **Exemplos de linguagens compiladas:** C, C++, Rust, COBOL e Pascal.

### Interpretação
As linguagens puramente compiladas tradicionais possuíam algumas desvantagens:
* **Falta de portabilidade:** Quando se compila um código em uma máquina, não há garantia de que ele rodará em outra com sistema diferente. Por exemplo, um executável compilado para Linux não roda nativamente no Windows.
* **Tempo de compilação:** Para sistemas muito grandes, o processo de compilação pode demorar minutos ou até horas, o que atrasa o ritmo de desenvolvimento.

As **linguagens interpretadas** mitigam esses problemas. Elas não geram um arquivo binário final antes de rodar. Em vez disso, um programa chamado *interpretador* lê o código linha por linha e o traduz em tempo real, executando os comandos na hora. Se a máquina de destino tiver o interpretador instalado, o código rodará nela sem problemas.
\
Mas nem tudo são flores! As linguagens interpretadas acabam sendo **mais lentas** que as compiladas, justamente pelo esforço extra de traduzir o código enquanto o programa está rodando.

> **Exemplos de linguagens interpretadas:** JavaScript, Python* e Ruby.  
> *(Nota: O Python moderno usa um modelo híbrido que gera arquivos `.pyc`, mas seu comportamento principal para o desenvolvedor é o de uma linguagem interpretada).*

### Intermediária (Linguagens de Máquina Virtual)
E se juntássemos o melhor dos dois mundos? Daí surgiram as linguagens que são compiladas para um formato **intermediário** (chamado de *Bytecode*) e, depois, esse formato é interpretado por uma máquina virtual. 
\
O Java popularizou esse conceito! Ele traz um desempenho muito maior do que as linguagens puramente interpretadas e, ao mesmo tempo, permite que o programa rode em qualquer sistema operacional que possua a sua máquina virtual (JVM), sem a necessidade de reescrever ou recompilar o código do zero.

> **Exemplos de linguagens intermediárias:** C#, Java e Kotlin.

## Fortemente Tipada x Fracamente Tipada

No nosso mundo, temos letras que representam palavras, números inteiros para contagens e números decimais para medições. Nas linguagens de programação, essas divisões também existem e são chamadas de **tipos**. A forma como a linguagem lida com a mistura desses tipos define se ela é forte ou fracamente tipada.

* **Fortemente tipada:** A linguagem não realiza conversões de tipos de maneira automática se isso puder gerar ambiguidade. Se você tentar somar o texto `"2"` com o número `2`, ela acusará um erro e exigirá que você converta explicitamente os valores.  
  *Exemplo:* Python.
* **Fracamente tipada:** A linguagem realiza conversões automáticas (coerção de tipos) para tentar fazer a operação funcionar, mesmo que o resultado seja confuso.  
  *Exemplo:* JavaScript (onde `"2" + 2` resulta no texto `"22"`).

## Tipagem Estática x Tipagem Dinâmica

Outra característica crucial sobre como as linguagens lidam com dados:

* **Tipagem Estática:** O tipo da variável é definido no momento em que ela é criada e **não pode ser alterado** durante a execução do programa. Se uma variável nasceu para guardar um número inteiro, ela morrerá guardando um número inteiro.  
  *Exemplo:* Java, C, C++.
* **Tipagem Dinâmica:** O tipo da variável é definido pelo valor que ela carrega no momento e **pode mudar** ao longo do programa. Uma variável pode começar guardando um texto e, linhas depois, receber um número sem gerar erros.  
  *Exemplo:* Python, JavaScript.

## Linguagens de Alto Nível e Baixo Nível

Por fim, classificamos as linguagens pela sua proximidade com a máquina ou com o ser humano:

* **Baixo Nível:** São linguagens que estão mais próximas da arquitetura física do computador. Suas instruções são diretas para o hardware. 
  * **Vantagens:** Altíssimo desempenho e controle total sobre a memória e componentes.
  * **Desvantagens:** Manutenção muito complexa, depuração difícil e uma **curva de aprendizado alta** (ou seja, são mais difíceis e demoradas de aprender).  
  *Exemplo:* Assembly.

* **Alto Nível:** São linguagens mais abstratas, que utilizam termos em inglês e estruturas lógicas muito próximas da nossa forma de pensar.
  * **Vantagens:** Código fácil de ler, manutenção mais simples, depuração amigável e uma **curva de aprendizado baixa/suave** (são mais fáceis de aprender).
  * **Desvantagens:** Podem ser mais lentas e consomem mais memória do que as de baixo nível.  
  *Exemplos:* Python, JavaScript, C#, Java.
